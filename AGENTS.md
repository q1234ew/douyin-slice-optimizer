# Douyin Slice Optimizer Agent Development Contract

适用范围：本文件所在仓库及全部子目录。

本文件是项目开发 Agent 的全局执行摘要。开始工作前必须阅读并遵守；详细产品口径、验收规则和当前状态分别以以下文档为准：

- `docs/product-goals.md`：最高层产品目标、指标和持续迭代原则。
- `docs/development-requirements.md`：工程、测试、数据、API、前端和 Definition of Done。
- `docs/current-state.md`：当前真实完成度、服务状态、已知缺口和下一步优先级。
- `docs/architecture.md`：模块边界、统一候选链路、数据模型和 API contract。
- `docs/model-and-algorithm-radar.md`：模型/算法机会、证据、成本和准入状态。

历史计划或实验报告与上述规范冲突时，不沿用过期结论。用户当前明确要求和更高层指令始终优先。

## 1. 北极星目标

所有工作必须明确对齐至少一个目标：

- **G1 已切短片筛选排名**：支持单条或批量导入已经剪好的短视频，保持原始边界，完成理解、分类、筛选、推荐排名、解释、审核和指标回流。
- **G2 完整节目智能切片**：对完整节目自动理解、召回、切片和分类，生成的候选进入 G1 的同一套排名、审核和反馈链路。
- **G3 本地/公网模型协同**：默认本地优先，允许通过可插拔 Provider 按需使用 Qwen、Kimi 等公网模型 API，但必须评估质量、费用、延迟、隐私、缓存和失败降级。

目标是提高在抖音等短视频平台获得优质流量、转发和关注的概率，不承诺爆款，不破解或模拟平台内部推荐系统，不刷量、不诱导互动、不绕过规则。

## 2. 当前主线优先级

除非用户明确调整优先级，按以下顺序判断工作价值：

1. 补齐 G1“原片段不可变”的批量短片直接排名入口。
2. 让 G1 与 G2 共用标准化候选、ranker、解释、质量 Gate、审核和指标回流 contract。
3. 建立完整节目召回 Gold 与统一候选排序的冻结 benchmark。
4. 补齐官方或授权账号的曝光、观看、分享和关注反馈闭环。
5. 在以上基础稳定后实现公网模型 Provider、预算台账和质量/成本对比。

研究支线不得因模型新颖而挤占产品主线。未通过门禁的功能保持 `research_only` 或 `shadow`，不得写生产排序权重。

## 3. 双入口、一条候选链路

已切短片和完整节目只允许在“候选如何产生”上不同：

```text
已切短片 ----------------------------> 标准化候选
完整节目 -> 理解 -> 召回 -> 切片/分类 -> 标准化候选
                                         -> 统一排序与解释
                                         -> 人工审核与导出
                                         -> 平台指标回流
```

禁止为两种入口复制两套逐渐漂移的评分规则。新增特征、模型分和质量 Gate 应进入统一 contract，并具有版本、来源、置信度和缺失/降级语义。

## 4. 每项任务开始前

1. 阅读与任务相关的规范文档和现有实现，不凭历史对话猜测当前状态。
2. 明确任务对齐的 G1/G2/G3、用户可感知成果、目标指标和非目标。
3. 检查工作区状态；仓库可能包含用户未提交改动，必须保留并避免覆盖无关文件。
4. 优先复用现有模块、contract、组件、脚本和测试，不平行创建重复体系。
5. 若需求是诊断或评审，只读取和报告；未经要求不扩大为重构、数据重建、发布或外部写操作。
6. 若合理假设不会改变任务方向，可继续执行并在交付中说明；会 materially 改变结果、成本或数据的选择必须向用户确认。

## 5. 产品与算法指标

不得只优化播放量。统一目标口径为：

```text
优质流量 + 观看质量 + 转发价值 + 关注转化
- 负反馈 - 合规风险 - 推理/API/人工成本
```

优先指标：

- 线上：曝光归一化播放、5 秒留存、平均观看比例、完播、复播、分享率、关注转化率、负反馈率。
- 代理：收藏、评论质量、点赞、可见互动热度、账号历史匹配。
- 离线：Recall@K、NDCG@K、Top-K lift、严重错判率、覆盖率、校准误差。
- 工程：单节目/单候选费用、GPU/CPU 时间、P50/P95 延迟、缓存命中率、失败率、人工复核时间。

播放或曝光缺失时只能写“高互动代理”或“研究结论”，不能把其他计数冒充播放量，也不能声称已经证明高流量。

## 6. 模型与算法准入

新增模型、提示词、规则或排序算法必须先说明：

- 解决的问题和当前基线。
- 冻结数据集、时间/账号/节目切分和防泄漏方式。
- 预期改善的指标及消融/对比方案。
- 延迟、显存/内存、API 费用、失败率和人工成本。
- 调用范围：全量、高价值候选、高分歧样本或高不确定样本。
- 超时、不可用、预算耗尽时的降级和旧版本回滚方式。
- 当前状态：`watch`、`validate`、`research_only`、`shadow`、`candidate`、`adopted` 或 `rejected`。

单条成功案例、模型排行榜或营销材料不能替代冻结 benchmark。未经验证的模型结果不能自动写人工 Gold、自动批准候选、直接替换生产排序或自动发布。

## 7. 公网模型 API 要求

- 公网模型默认关闭，只有用户配置 Provider、密钥、预算和数据许可后才启用。
- 使用厂商无关 Provider contract；业务代码不直接绑定单一厂商返回结构。
- API key 仅从环境变量或密钥服务读取，不写数据库、日志、缓存、报告或前端 bundle。
- 记录厂商、模型/API 版本、提示词版本、输入规模、输出 token、请求数、重试、限流、延迟、缓存命中和估算费用。
- 支持单请求、单批次和单日预算；达到上限停止新增请求并回退本地结果。
- 优先上传代表帧、短音频窗口或结构化摘要，不默认上传完整节目。
- 相同内容 hash、模型、提示词和参数应命中缓存，避免重复计费。
- 外部响应视为不可信输入，必须做 schema 校验、超时、重试上限、回显/幻觉过滤和失败隔离。
- 涉及模型能力、定价或接口限制的时效性判断，应核对厂商官方文档并注明查询日期。

## 8. 数据与评测要求

- 训练主数据使用自有、授权或许可来源；公开/可见数据按现有合规口径用于研究和弱先验。
- 保留来源、账号、节目、平台 item ID、内容 hash、数据集版本和标签窗口。
- 同源重复片段不得跨训练/验证；优先采用时间切分、账号隔离或节目隔离。
- 召回与排序分开评估：完整节目候选召回看 Recall@K，标准化候选排序看 NDCG@K/Top-K lift。
- 冻结 benchmark、manifest 和历史报告不可原地覆盖；新版本生成新 ID，并记录代码/模型/提示词版本。
- 修改数据库、重导数据或覆盖 transcript/candidate 前必须按开发要求备份；未经用户授权不运行破坏性批处理。
- 缺失值、弃权、fallback 和模型失败必须显式记录，不得用零分或伪模型结果掩盖。

## 9. 工程实现约束

- 后端继续复用 Python、FastAPI、SQLite 和现有 service/feature/learning 分层。
- 前端只修改 `frontend/src/`，使用现有 Vue 3、Vite、TypeScript 和 lucide-vue 体系；静态构建产物由构建命令生成。
- 新 API 应有稳定 contract/version、可操作错误信息以及 CLI、前端或文档入口。
- 长任务、GPU 推理和公网调用应支持状态、超时、缓存、断点/重试或明确的同步边界，不能阻塞健康检查。
- 16GB GPU 上的大模型默认串行加载；禁止无资源门控同时驻留造成 OOM。
- 模型不可用时保留确定性规则或本地基线；不得因可选能力失败拖垮主流程。
- 不手工编辑生成目录，不提交密钥、大型模型权重、临时媒体、运行缓存或无关产物。

## 10. 变更与验证

按风险执行最小充分验证：

- 后端/算法：编译检查、相关单元测试；影响共享 contract 时运行更广回归。
- 排序/模型：相关测试外，还要提供冻结 benchmark、基线比较和质量/成本报告。
- 前端：类型检查或生产构建，并用浏览器验证受影响流程、控制台和响应式布局。
- 数据库：先备份，再检查迁移、计数、去重和向后兼容。
- 部署/服务：检查 health、模型实际加载状态、真实最小输入和最终默认运行态。
- 文档-only：不要求完整测试，但必须检查 Markdown、相对链接、命令和当前状态数字。

测试失败不得被隐藏。若失败来自既有环境或无关改动，应给出命令、错误和影响范围。

## 11. 文档与主动汇报

完成以下变化时同步维护：

- 产品目标、指标或长期边界：`docs/product-goals.md`
- 当前能力、数据规模、服务状态或下一步：`docs/current-state.md`
- 工程、测试、数据或部署规则：`docs/development-requirements.md`
- 模块、数据模型或 API contract：`docs/architecture.md`
- 新模型/算法发现、benchmark 或准入状态：`docs/model-and-algorithm-radar.md`
- 用户操作、CLI 或页面流程：`docs/user-manual.md`

开发中发现可能提升切片召回、排序质量、流量、转发、关注或成本效率的新算法/模型时，必须：

1. 更新模型与算法雷达。
2. 在当次交付中主动告知用户，不等待追问。
3. 摘要说明问题、证据、预期收益、成本、风险、最小验证和建议状态。
4. 未完成冻结 benchmark 时明确标记为研究假设，不写成已证实收益。

## 12. Definition of Done

任务完成前确认：

- 已说明对齐的 G1/G2/G3 和用户可见结果。
- 实现复用统一候选与排序 contract，没有制造重复主链路。
- 相关测试、构建、真实探针或文档检查已通过；未执行项有明确理由。
- 数据、模型、排序权重、人工 Gold、导出和发布行为的变化已说明。
- 新增计算/API/人工成本及失败降级方式已说明。
- 相关长期文档已同步，发现的新模型/算法已登记并主动汇报。
- 未覆盖用户已有改动，未留下失控服务、错误默认模型或未说明的破坏性状态。
