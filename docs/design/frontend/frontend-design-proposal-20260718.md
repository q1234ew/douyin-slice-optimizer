# 前端设计方案

日期：2026-07-18

适用范围：Douyin Slice Optimizer Web 工作台现状评审与下一轮前端重构

目标对齐：

- G1 已切短片筛选排名
- G2 完整节目智能切片与统一候选审核

本方案基于四类证据整理：

- 当前仓库前端代码结构与状态管理
- 本地实际运行中的工作台页面与真实数据状态
- 当前产品目标与工程状态文档
- 既有 UX 审核结论的交叉验证

## 方案结论

当前前端已经具备成熟的深色专业工具基调，真实问题不在“风格不够好看”，而在“任务层级和信息优先级不够稳定”。页面已经能支撑节目处理、候选审核、研究评测三类核心工作，但高频生产流和低频研究流仍然被放在过近的层级，导致首屏重复解释、局部过载、决策动作不够聚焦。

下一轮前端设计建议不做大面积视觉推翻，而是做一次以任务流为中心的信息架构重构：

- 保留当前深色工作台视觉语言
- 强化“剪辑生产流”和“研究运营流”的边界
- 把质量哨兵从大块报告改成压缩结论层
- 把推荐模拟并入候选判断依据，而不是独立强曝光页面
- 把候选卡和详情改成“默认给结论，按需展开证据”
- 把研究中心拆成更清晰的二级任务空间

## 当前真实前端结构

当前工作台的一级结构是：

- 顶栏一级模式：`剪辑工作台`、`研究中心`
- 剪辑工作台内部分页：`素材与节目`、`候选审核`、`发布准备`
- 素材入口双模式：`已切短片`、`完整节目`
- 候选详情分区：`决策`、`字幕/ASR`、`历史先验`、`包装/平台`
- 研究中心分区：`概览`、`样本与标注`、`评测与校准`、`平台连接`、`模型与环境`

当前真实页面状态中可以确认：

- 默认首页优先落在 `已切短片排名`，符合 G1 主线优先级
- `完整节目` 模式已具备节目列表、状态筛选、分步处理、质量哨兵和一键智能切片
- `候选审核` 已形成稳定的“候选列表 + 右侧详情”结构，并能联动导出、ASR verify 和判断依据
- `研究中心` 已承载数据口径、账号质量、授权指标导入、抖音连接、语义校准、算法回测和运行环境

这说明系统不是“功能不完整”，而是已经进入“需要提升可理解性和操作效率”的阶段。

## 当前设计问题

### 1. 首屏被系统说明占据过多

在 `完整节目` 模式下，当前页面首屏同时出现：

- 左侧流程导航
- 中间下一步任务
- 当前节目信息
- 顶部统计
- 质量哨兵
- 下方节目列表

问题不是信息不重要，而是这些信息都被放在同等优先级上。用户真正要处理的对象是“当前节目”和“下一步动作”，但现在首屏更像系统状态总览。

影响：

- 节目列表被压到下方
- 高价值动作需要二次扫描才能找到
- 页面开始阶段的认知成本偏高

### 2. 质量哨兵在多个页面重复出现

质量哨兵当前同时出现在：

- 节目处理页
- 候选审核页
- 推荐模拟页

它的价值很高，但表现层级过重。用户在不同页面里会反复看到同一套风险摘要、统计卡和“开始复核 / 查看报告”动作，导致系统提示挤压业务主体内容。

建议判断：

- 质量哨兵应保留
- 但默认应压缩为一条结论型风险横幅
- 详细报告只在需要诊断时展开

### 3. 候选审核已接近主工作区形态，但卡片信息过载

候选页是当前最成熟的一页，但单卡同时承载：

- 时间段
- 标题
- 摘要文案
- 分数
- Omni 状态
- 风险状态
- 导出状态
- 查看详情动作

这让“扫一眼决定先看哪条”变慢。用户第一需求其实是：

- 这条值不值得看
- 现在该做什么
- 为什么它排在前面

而不是先理解全部技术维度。

### 4. 候选详情是好结构，但标签平级关系还不够自然

当前详情标签为：

- 决策
- 字幕/ASR
- 历史先验
- 包装/平台

这四个标签实际上属于不同任务阶段：

- 决策：审核阶段
- 字幕/ASR：修复阶段
- 历史先验：解释阶段
- 包装/平台：发布阶段

放在同一个平级标签条里没有错，但会削弱“当前该做哪一步”的顺序感。

### 5. 推荐模拟的价值更像判断依据，不像一级独立页面

从功能上看，推荐模拟回答的是：

- 这条候选在推荐链路里会卡在哪
- 哪些调整可能提升表现
- 当前候选和其他候选的瓶颈差异是什么

这更接近候选判断依据，而不是一个需要单独跳出的工作台页面。独立成页后，用户会先看到与节目处理页高度相似的风险信息，真正的模拟价值反而被稀释。

### 6. 研究中心功能强，但承载过重

研究中心当前已经是一个复合型专业后台，包含：

- 数据健康
- 样本资产
- 平台回流
- 语义校准
- 素材审核
- 算法回测
- 模型环境

问题不是功能太多，而是它们都被包在一个大页签容器里。对于高频研究用户来说，这会带来：

- 路径过深
- 完成态空白较多
- 操作区与结果区切换成本高

## 设计目标

下一轮重构建议聚焦五个目标：

1. 让高频生产任务在 3 秒内识别“当前对象、当前风险、下一步动作”。
2. 让候选审核支持更快的扫读与批量判断。
3. 让系统解释层后退，用户决策层前置。
4. 让研究中心从“大杂烩页签”进化为“清晰的研究任务空间”。
5. 在不推翻现有代码结构的前提下，优先通过信息重排、组件收敛和交互分层提升体验。

## 新信息架构建议

建议把产品分成两大母空间：

```text
剪辑工作台
├── 素材入口
├── 节目处理
├── 候选审核
├── 导出与回流

研究中心
├── 研究概览
├── 样本与标注
├── 评测与校准
├── 平台连接
├── 模型与环境
```

具体调整：

- 保留顶栏一级导航：`剪辑工作台`、`研究中心`
- 保留工作台内部阶段导航，但只表达生产阶段，不再混入研究动作
- `推荐模拟` 从独立页退居为候选详情中的 `判断依据` 子模块
- 顶部抖音连接按钮仅保留状态提示，详情统一进入 `研究中心 > 平台连接`

## 页面级设计方案

### A. 素材与节目

目标：让用户快速知道“处理哪个对象”“卡在哪一步”“下一步点哪里”。

建议结构：

1. 顶部任务条
2. 当前对象卡
3. 压缩风险横幅
4. 节目列表
5. 展开的系统报告

重构建议：

- 左侧阶段条只保留阶段导航、异常数、整体完成度
- 中间首屏只保留当前节目卡和主按钮
- 质量哨兵压缩成一行：
  - `质量 57`
  - `3 类风险`
  - `10 条需复核`
  - 主按钮 `开始复核`
  - 次按钮 `查看报告`
- 节目列表上移到首屏
- 英文内部状态改成统一中文状态
- 已切短片和完整节目沿用同一页面骨架，仅替换列表字段和主要动作

### B. 已切短片排名

这一页其实已经非常接近产品主线表达，建议重点提升批量任务感：

- 批次头部增加统一任务状态：`上传中 / 提取中 / 排名中 / 已完成 / 部分失败`
- 强化批次层级而非单条层级
- 排名表默认突出：
  - 排名
  - 标题
  - 边界锁定
  - 统一排序分
  - 当前状态
  - 审核入口
- 将“历史证据高/中/低”等技术词改为更可操作的结论词，如：
  - `证据强`
  - `证据一般`
  - `待补证据`

### C. 候选审核

目标：让用户在最短时间内做出“通过 / 需修复 / 暂缓 / 导出”的判断。

建议布局保持 `列表 + 右侧详情`，但调整信息层级。

候选卡默认展示：

- 时间范围
- 标题
- 一句核心理由
- 分数
- 决策状态
- 主动作

候选卡隐藏到展开态或右侧详情：

- 长字幕摘要
- 结构、情绪、类型标签
- 详细模型说明
- 次级技术状态

状态体系建议统一为：

- 待复核
- 需修复
- 可导出
- 已导出
- 暂缓

质量风险不再和导出状态并列展示多个徽标，而作为“状态原因”补充说明。

批量能力建议补齐：

- 多选
- 仅看风险项
- 仅看未处理
- 批量通过
- 批量导出
- 批量暂缓

### D. 候选详情

建议把当前详情重新组织成“顶部固定决策区 + 下方证据区”。

顶部固定区：

- 候选标题
- 时间范围
- 当前状态
- 风险结论
- 主操作

下方证据区拆为三个主分区：

- 决策摘要
- 文本与内容校验
- 判断依据

对应原有内容映射：

- `决策` 保留为决策摘要
- `字幕/ASR` 保留，但改名为 `文本校验`
- `历史先验 + 推荐模拟` 合并为 `判断依据`
- `包装/平台` 仅对通过候选开放

这样更符合实际操作顺序。

### E. 推荐模拟

建议取消它的一级强曝光地位，改为两种存在方式：

- 作为候选详情中的 `判断依据` 子卡片
- 作为多候选对比工具，在需要时弹出

推荐模拟首屏不再复用完整质量哨兵，只保留一条轻量提示：

- `当前节目存在 2 类质量风险，模拟结果仅供参考`

核心内容改为对比结构：

- 原分
- 模拟分
- 主瓶颈
- 影响阶段
- 调整建议

### F. 研究中心

研究中心建议从“大页签”向“任务面板化”演进。

短期不必改成全新路由，但可以先做以下调整：

- `概览` 只放异常、变化和待处理事项
- `样本与标注` 聚焦数据质量与样本入口
- `评测与校准` 强化模式切换后的结果区，不让完成态留下大片空白
- `平台连接` 承接全部授权、导入与账号同步
- `模型与环境` 完全技术化，不与业务结论混排

其中 `评测与校准` 建议优先改造，因为它现在最像“页面里再嵌一个系统”。

## 视觉与交互原则

### 1. 保留现有深色专业工作台基调

当前视觉语言已经有几个正确方向：

- 深色背景适合长时间高密度工作
- 洋红与蓝色双强调色能区分动作与系统提示
- 卡片、标签、状态、边框语言已经较一致

因此不建议推翻当前视觉风格，只建议做三类收敛：

- 降低重复高亮
- 提高结论层对比
- 弱化技术说明的默认可见度

### 2. 统一“结论层 / 证据层 / 系统层”

所有页面的信息都应该按这三层组织：

- 结论层：现在该做什么
- 证据层：为什么这么做
- 系统层：模型、报告、contract、运行状态

默认只展开结论层和必要证据层。

### 3. 控制首屏组件数

同一首屏建议最多同时强调：

- 一个主对象
- 一个主结论
- 一个主动作
- 一个次级入口

超过这个数量，页面就会重新回到“系统说明面板”的感觉。

## 响应式建议

当前代码已经有较完整的响应式断点，但产品层面还可以再明确：

- `1120px` 以下：侧栏折叠为顶部阶段条
- `840px` 以下：候选详情改为抽屉或独立详情层
- `700px` 以下：素材入口与研究中心优先展示当前任务，不先展示完整导航

核心原则：

- 小屏优先展示当前对象和主操作
- 说明、报告、阶段全局导航改为折叠式

## 实施优先级

### P0

- 压缩质量哨兵为结论横幅
- 优化素材与节目首屏结构
- 精简候选卡信息密度
- 统一候选状态模型

### P1

- 合并推荐模拟到候选判断依据
- 重组候选详情标签结构
- 优化研究中心概览与评测页的信息层级
- 增加候选列表筛选与批量动作

### P2

- 研究中心二级任务空间化
- 批量任务中心
- 更强的多候选对比
- 小屏独立详情抽屉

## 对研发实现的建议

本轮更适合做“结构重排”和“组件收敛”，不建议一开始就大规模重写状态层。

建议实现顺序：

1. 先重构 `QualitySentinel` 的展示层级
2. 再整理 `ProgramWorkbench` 与 `WorkflowSidebar` 的首屏结构
3. 然后精简 `CandidateWorkbench` 卡片结构
4. 最后重组 `InspectorPanel` 与 `SimulationView` 的角色边界

这样可以在不打断现有 API contract 的前提下逐步落地。

## 最终建议

建议把下一轮前端目标定义为：

“在不改变核心能力与 API contract 的前提下，把当前工作台从算法能力展示界面，提升为以任务决策为中心的剪辑工作台。”

如果只做一句最关键的设计判断，那就是：

当前系统已经足够强，下一步不该继续增加首屏信息，而应该让用户更快看见结论、更少读取系统解释、更顺地完成审核与导出闭环。
