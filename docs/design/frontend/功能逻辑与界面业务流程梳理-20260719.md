# 功能逻辑与界面业务流程梳理

日期：2026-07-19

适用对象：

- 产品梳理
- 前端改版对齐
- 设计稿继续收敛
- 后续真实前端落地

## 文档目的

当前页面稿已经基本覆盖主要界面，但要继续推进设计和前端改造，必须先把三件事对齐：

- 功能模块到底在解决什么问题
- 页面之间真实的业务流转关系是什么
- 用户在每一步到底“先看什么、再做什么、最后去哪”

这份文档重点不是讨论视觉，而是把产品的使用逻辑梳理清楚。

## 一、产品的核心目标

这个系统本质上不是一个通用内容后台，而是一个围绕短视频切片优化的专业工作台。

它的主目标有两条：

1. 把完整节目或已切短片，转成可发布的高质量候选短视频。
2. 用研究与评测能力持续修正排序、校准、平台连接和模型环境，保证主链路稳定。

因此整个平台天然分成两个母空间：

- `剪辑工作台`：负责生产与决策闭环
- `研究中心`：负责支撑、校准、诊断与连接

## 二、系统的顶层信息架构

建议把整套产品统一理解为下面的结构：

```text
剪辑工作台
├── 素材与节目
├── 候选审核
└── 发布准备

研究中心
├── 概览
├── 样本与标注
├── 评测与校准
├── 平台连接
└── 模型与环境
```

这里有一个很关键的原则：

- `剪辑工作台` 解决“做什么”
- `研究中心` 解决“为什么这么做、现在是否可靠、哪里出问题了”

也就是说，两者不是平级的“两个后台菜单”，而是：

- 一个负责主流程执行
- 一个负责主流程稳定性与方法论支撑

## 三、主业务流程

### 1. 生产主链路

这是用户最高频、最核心的流程：

```text
素材与节目
→ 候选审核
→ 发布准备
→ 导出 / 发布
```

这条链路对应的是：

- 找到当前要处理的对象
- 生成或接收候选
- 判断候选是否值得发
- 处理包装与导出

### 2. 研究支撑链路

这是支撑主链路稳定性的后台流程：

```text
研究中心概览
→ 样本与标注
→ 评测与校准
→ 平台连接
→ 模型与环境
```

它不直接产出可发布视频，但会影响：

- 候选排序是否可信
- 校准偏差是否可接受
- 授权和指标回流是否完整
- 模型与预算是否还能支撑主链路

## 四、各页面真正承担的职责

### A. 素材与节目

这是主链路的入口页，负责解决三个问题：

1. 当前处理对象是谁
2. 当前卡在哪一步
3. 下一步应该点什么

它不应该承担太重的系统解释任务，也不应该一上来塞满研究信息。

它的业务职责应该是：

- 选择处理对象
- 展示当前节目或批次状态
- 呈现压缩后的风险结论
- 提供下一步动作入口

对用户来说，这一页的使用逻辑应该是：

1. 先确认当前节目或批次
2. 看当前是否有风险或阻塞
3. 直接进入下一步处理动作

### B. 候选审核

这是整套系统里最关键的决策页。

它真正负责的是：

- 判断候选值不值得进入发布准备
- 判断当前候选是否需要修整
- 理解候选为什么排在前面

这页不应该让用户先去理解一堆技术字段，而应该先让用户回答三个问题：

1. 这条能不能发
2. 不发的原因是什么
3. 如果能发，下一步是否直接进入发布准备

所以它的结构逻辑应该稳定为：

- 左侧：候选优先级与筛选
- 中间：当前候选的主决策
- 右侧：证据、字幕、风险、标题建议

### C. 发布准备

这是“审核完成之后”的收口页。

它负责把“可发布”变成“可交付”。

核心职责是：

- 标题建议与封面版本管理
- 9:16 成片规格确认
- 字幕与口播检查
- 发布前 checklist
- 导出动作

用户到这一页时，不再关注“要不要发”，而是只关注：

1. 怎么发得更稳
2. 哪些风险还没清空
3. 现在能不能导出

### D. 研究中心概览

研究中心概览不是单纯的页签入口，而应该是研究域的总控页。

它负责给出：

- 当前最重要的异常
- 当前最应该处理的研究事项
- 各支撑链路当前是否在线
- 最近关键变化

它真正回答的是：

- 研究系统现在健康吗
- 哪个问题最优先
- 现在应该先处理哪里

### E. 评测与校准

这一页不是为了“展示很多指标”，而是为了支持判断。

它要回答：

1. 当前模型和排序是否可靠
2. 哪些分段或样本出现了校准偏差
3. 现在应该继续放量、保持观察，还是回滚

所以这里应该把数据组织成：

- benchmark 当前状态
- 模型对比
- 校准偏差
- 严重错判
- 最近回测任务
- 结论建议

### F. 平台连接与模型环境

这一页不是纯设置页，而是连接与运维工作台。

它真正负责的是：

- 平台授权是否在线
- 指标回流是否稳定
- Provider 是否可用
- 预算是否受控
- 环境是否正常

它的输出应该是可判断的状态，而不是一堆配置项。

## 五、用户使用逻辑

### 1. 典型高频用户路径

一个典型用户进入系统后的真实路径应该是：

1. 打开 `素材与节目`
2. 选择今天要处理的节目或批次
3. 看到压缩后的风险结论
4. 进入 `候选审核`
5. 快速完成通过 / 修整 / 暂缓判断
6. 进入 `发布准备`
7. 完成标题、封面、字幕、导出前检查
8. 导出成片

这是日常生产主链路。

### 2. 典型异常排查路径

当用户发现结果不对或系统不稳定时，使用逻辑应切换到：

1. 从工作台察觉异常
2. 进入 `研究中心概览`
3. 看当前关键异常
4. 进入 `评测与校准` 或 `平台连接`
5. 确认是排序偏差、授权问题、预算问题还是环境问题
6. 回到主工作台继续处理

这条链路强调的是“定位原因”，不是“继续生产”。

## 六、页面之间的业务跳转关系

### 剪辑工作台内部跳转

`素材与节目 → 候选审核`

触发条件：

- 节目已生成候选
- 已切短片已完成统一排序
- 需要人工判断

`候选审核 → 发布准备`

触发条件：

- 当前候选被人工通过
- 或达到可导出状态

`发布准备 → 导出`

触发条件：

- checklist 通过
- 标题、封面、字幕、规格达到可交付状态

### 研究中心与工作台之间的跳转

`工作台 → 研究中心`

触发条件：

- 用户对当前候选或排序结果不确定
- 发现质量偏差
- 发现指标回流或模型异常

`研究中心 → 工作台`

触发条件：

- 支撑问题已确认
- 当前结论足够恢复主链路处理

## 七、关键状态系统

如果后续要让前端和设计更稳定，状态系统必须统一。

### 1. 节目 / 批次状态

建议统一为：

- `待处理`
- `已转写`
- `已生成候选`
- `已评分`
- `待复核`
- `部分异常`
- `已完成`

### 2. 候选状态

建议统一为：

- `待审核`
- `建议放行`
- `需修整`
- `继续观察`
- `确认通过`
- `进入发布准备`
- `已导出`

### 3. 系统 / 研究状态

建议统一为：

- `稳定在线`
- `可继续`
- `持续观察`
- `待处理`
- `受控放行`
- `挂起中`
- `已关闭`

状态命名的原则是：

- 先表达业务判断
- 再表达技术含义
- 不让用户先看到内部术语

## 八、当前最容易混乱的几个点

### 1. “模式”和“阶段”容易混

顶部双层导航必须明确回答两个不同问题：

- 第一层：我在哪个工作模式
- 第二层：我在这个模式里的哪个阶段

如果两层都长得像按钮组，用户会混淆。

### 2. 工作台页和研究页语气容易分裂

工作台语气应该更偏：

- 当前任务
- 当前对象
- 当前动作

研究页语气应该更偏：

- 当前异常
- 当前判断
- 当前支撑状态

但它们要属于同一个产品，所以视觉语言可以统一，信息重心不能混。

### 3. 说明文案容易过多

当前最容易出现的问题是：

- 标题下再加一段解释
- 卡片里再加一段方法说明
- 标签里还承担说明作用

正确做法应该是：

- 页面标题只说页面身份
- 状态标签只说状态
- 说明只在真正需要辅助判断时出现

## 九、设计与前端后续该怎么对齐

如果继续推进，这里建议分成三层来做：

### 第一层：逻辑对齐

- 统一模块职责
- 统一状态体系
- 统一页面间跳转逻辑

### 第二层：界面结构对齐

- 统一页头与导航
- 统一工作台结构
- 统一研究页结构

### 第三层：组件对齐

- 工具栏
- 列表行
- 详情面板
- 状态标签
- 摘要卡
- 主次按钮

## 十、建议的下一步

如果继续往下推进，建议直接做下面三个输出：

1. `页面功能结构图`
2. `页面跳转与状态流转图`
3. `前端落地改造优先级清单`

这样后续无论是继续打磨设计，还是正式改真实前端，都不会再在“页面到底干什么、为什么要这样跳”上反复回头。

## 结论

当前这套产品最重要的整合思路可以概括成一句话：

把“剪辑工作台”定义为生产闭环，把“研究中心”定义为支撑闭环，再把所有页面都压回到“当前对象、当前判断、下一步动作”这三个问题上。

只要这个逻辑统一，后续无论是设计打磨还是前端重构，都会稳定很多。
