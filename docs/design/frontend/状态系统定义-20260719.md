# 状态系统定义

日期：2026-07-19

适用对象：

- 产品经理
- 设计师
- 前端开发
- 测试

## 文档目的

当前系统已经有比较完整的页面结构，但状态表达仍然容易漂移：

- 同一类状态在不同页叫法不同
- 有些状态偏业务，有些状态偏技术
- 有些标签承担了过多说明功能
- 颜色和语气没有完全统一

这份文档用于统一：

- 状态命名
- 状态层级
- 状态颜色语义
- 状态使用场景
- 状态在各页面的展示方式

## 一、状态系统总原则

### 1. 先表达业务判断，再表达技术含义

状态首先要让用户知道：

- 现在是否正常
- 是否要处理
- 是否能继续下一步

而不是先告诉用户内部系统细节。

例如：

- 优先使用 `待审核`
- 不优先使用 `pending_review_v2`

### 2. 标签只表达状态，不承担说明段落

状态标签应该短、稳、可扫描。

状态标签不适合承担：

- 方法解释
- 背景说明
- 决策建议全文

例如：

- 正确：`待处理`
- 不建议：`当前建议优先处理这项问题`

### 3. 一个状态只表达一个主判断

避免一个标签同时表达多个信息维度。

例如：

- `待处理`
- `持续观察`
- `稳定在线`

不要用：

- `轻微异常但可继续`

这种复合语义应该拆成：

- 主状态
- 次说明

### 4. 同一状态跨页尽量同名

例如：

- 工作台页、研究页都出现“需要处理”的情况时，优先统一为 `待处理`

不要一页写：

- `待处理`

另一页写：

- `待跟进`

再另一页写：

- `待推进`

除非它们在业务含义上确实不同。

## 二、状态分层

建议把全系统状态分成四层：

### L1：流程状态

用于表示对象在业务流程中的位置。

例如：

- `待处理`
- `已转写`
- `已生成候选`
- `待审核`
- `进入发布准备`
- `已导出`

### L2：判断状态

用于表示当前页面里的人工或系统判断。

例如：

- `建议放行`
- `需修整`
- `继续观察`
- `确认通过`
- `待处理`
- `可继续`

### L3：系统状态

用于表示研究、连接、环境、Provider 之类的运行状态。

例如：

- `稳定在线`
- `受控放行`
- `挂起中`
- `已关闭`

### L4：风险状态

用于表示风险程度或注意级别。

例如：

- `局部风险`
- `持续观察`
- `部分异常`

注意：

- 风险状态不应该替代流程状态
- 风险状态通常是补充，而不是主状态

## 三、状态颜色语义

建议统一为六类语义色，而不是按页面随意定义。

### 1. 中性 `neutral`

适用：

- 尚未开始
- 普通说明
- 没有明显正负倾向

典型状态：

- `待处理`
- `待审核`
- `排队中`
- `继续观察`

视觉建议：

- 低对比中性底
- 中性边框
- 次级文字色

### 2. 主流程 `primary`

适用：

- 当前态
- 当前主阶段
- 当前焦点
- 主链路中的确认状态

典型状态：

- `当前阶段`
- `当前焦点`
- `建议放行`
- `进入发布准备`

视觉建议：

- 使用主色弱底
- 控制使用频率，避免全屏蓝色标签

### 3. 成功 `success`

适用：

- 已完成
- 可继续
- 稳定运行

典型状态：

- `已完成`
- `确认通过`
- `稳定在线`
- `可继续`
- `字幕校验通过`

视觉建议：

- 绿色语义
- 不要过亮，不做强烈警报式表现

### 4. 观察 `warning`

适用：

- 需要关注
- 还未阻塞
- 需要继续观察

典型状态：

- `持续观察`
- `观察中`
- `待续签`
- `继续观察`

视觉建议：

- 暖色弱底
- 不直接等同错误

### 5. 风险 / 异常 `danger`

适用：

- 明确异常
- 明确阻塞
- 明确需要立即处理

典型状态：

- `局部风险`
- `需修整`
- `部分异常`
- `挂起中`
- `待处理`

注意：

- `待处理` 只有在明确阻塞场景下才可用 danger
- 一般页面中的 `待处理` 默认仍建议先用 neutral

### 6. 关闭 / 停用 `disabled`

适用：

- 当前关闭
- 不参与主链路
- 只作为影子态存在

典型状态：

- `已关闭`
- `影子验证`

视觉建议：

- 更浅的中性灰
- 不应引导成为主视觉焦点

## 四、业务对象状态定义

## 1. 节目 / 批次状态

适用对象：

- 完整节目
- 已切短片批次
- 节目处理队列

建议状态定义如下：

### `待处理`

含义：

- 节目或批次已进入系统，但还没开始当前链路处理

展示建议：

- 中性状态

### `已转写`

含义：

- 文本基础已具备，可以继续进入候选生成或评分

展示建议：

- 中性偏完成

### `已生成候选`

含义：

- 候选已经召回，但尚未进入完整人工判断

展示建议：

- 主流程或中性状态

### `已评分`

含义：

- 系统评分已完成，可进入人工审核

展示建议：

- 主流程状态

### `待复核`

含义：

- 当前对象已经到了需要人工确认的节点

展示建议：

- 主流程重点状态

### `部分异常`

含义：

- 当前对象可继续，但链路里有部分问题需要回看

展示建议：

- 风险状态

### `已完成`

含义：

- 当前页面职责内的流程已走完

展示建议：

- 成功状态

## 2. 候选状态

适用对象：

- 候选卡
- 候选列表
- 候选详情
- 发布准备入口

建议状态定义如下：

### `待审核`

含义：

- 还未做明确人工判断

### `建议放行`

含义：

- 系统或当前判断倾向于通过，但尚未最终确认

### `需修整`

含义：

- 候选值得保留，但需要修边界、修口播、修标题等

### `继续观察`

含义：

- 当前不立即通过，但保留候选资格

### `确认通过`

含义：

- 已完成人工确认，可进入后续包装

### `进入发布准备`

含义：

- 候选已经开始进入包装与导出前处理

### `已导出`

含义：

- 当前候选已完成导出动作

## 3. 研究与系统状态

适用对象：

- 研究中心概览
- 评测与校准
- 平台连接
- 模型与环境

建议状态定义如下：

### `稳定在线`

含义：

- 当前支撑能力正常

### `可继续`

含义：

- 当前有小问题但不阻塞主链路

### `持续观察`

含义：

- 需要关注趋势，但还不需要立即干预

### `待处理`

含义：

- 已确认需要有人处理

### `受控放行`

含义：

- 当前能力可继续使用，但处于限制或保护条件下

### `挂起中`

含义：

- 当前流程被主动暂停，等待问题解决

### `已关闭`

含义：

- 当前能力不参与主链路

## 五、页面级状态使用规范

## 1. 素材与节目

主状态应围绕：

- 节目 / 批次状态
- 风险结论
- 下一步动作

不建议在这一页同时混入大量系统研究状态。

### 推荐状态组合

- `待复核` + `部分异常`
- `已评分` + `可继续`
- `已完成`

## 2. 候选审核

主状态应围绕：

- 候选判断状态
- 修整状态
- 进入发布准备的资格

### 推荐状态组合

- `待审核`
- `建议放行`
- `需修整`
- `继续观察`
- `确认通过`

## 3. 发布准备

主状态应围绕：

- 包装是否完成
- 风险是否清空
- 是否可以导出

### 推荐状态组合

- `待处理`
- `可继续`
- `持续观察`
- `已导出`

## 4. 研究中心

主状态应围绕：

- 当前是否阻塞
- 当前是否在线
- 当前是否需要处理

### 推荐状态组合

- `待处理`
- `持续观察`
- `稳定在线`
- `受控放行`
- `挂起中`

## 六、文案规范

### 推荐写法

- `待审核`
- `建议放行`
- `需修整`
- `持续观察`
- `稳定在线`
- `受控放行`

### 不推荐写法

- `当前建议优先通过`
- `需要进一步人工再次观察`
- `目前系统整体仍然可以继续`
- `存在轻度异常但不影响主链路`

原因：

- 太长
- 扫描效率低
- 很难跨页统一

## 七、前端实现建议

建议把状态实现分成两层：

### 1. 语义层

例如：

- `statusType = neutral | primary | success | warning | danger | disabled`

### 2. 文案层

例如：

- `label = 待审核`
- `label = 稳定在线`

这样做的好处是：

- 视觉和文案解耦
- 后续可统一替换颜色和样式
- 同一语义下能复用不同业务状态名称

## 八、建议的统一映射

### 业务状态到语义色映射

| 业务状态 | 语义色 |
| --- | --- |
| 待处理 | neutral / danger |
| 待审核 | neutral |
| 建议放行 | primary |
| 需修整 | danger |
| 继续观察 | warning |
| 确认通过 | success |
| 已导出 | success |
| 稳定在线 | success |
| 可继续 | success |
| 持续观察 | warning |
| 受控放行 | primary |
| 挂起中 | danger |
| 已关闭 | disabled |

注：

- `待处理` 在普通列表里建议用 `neutral`
- `待处理` 在关键阻塞卡里可以升级为 `danger`

## 九、建议的下一步

如果前端开始真实改造，建议先做：

1. 状态枚举统一
2. 状态标签组件统一
3. 各页面旧状态词替换映射表

## 结论

状态系统的目标不是“给页面上很多彩色小标签”，而是让用户在最短时间内读懂：

- 现在是什么状态
- 这个状态意味着什么
- 下一步该做什么

只要状态命名、颜色语义和使用场景统一，整套产品的理解成本会立刻下降。
