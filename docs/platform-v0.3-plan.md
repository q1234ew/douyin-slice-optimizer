## 1. 当前多 Agent 进展

### 1.1 Web UI / 推荐模拟

主线已经新增推荐链路模拟器：

- `src/dso/simulation/recommender.py`
- `GET /videos/{video_id}/simulation`
- `GET /segments/{segment_id}/simulation`
- Dashboard 顶部新增“推荐模拟”Tab。

这个模块将单条切片拆成冷启动、首轮留存、扩量互动、长尾消费、风险重排和队列多样性 6 个阶段，适合回答“这条切片为什么可能起量，卡在哪一层”。

### 1.2 ASR / GPU / whisper.cpp

ASR Agent 已完成真实样本转写与对比：

- faster-whisper 全量 CPU 跑通，但耗时较长。
- whisper.cpp 速度明显更好，适合做默认高速后端。
- 当前 `whisper_cpp:base` 仍会出现音乐/静音段重复幻觉、英文歌词偏置和广告口播混入问题。

结论：whisper.cpp 可以作为速度路径，但不能直接作为无护栏的发布依据。需要 VAD/静音过滤、重复文本检测、广告词降权和闭环候选复核。

### 1.3 产品工作台

当前 Dashboard 已具备：

- 视频列表和运行环境状态。
- 候选切片列表。
- 评分详情、标题建议、封面建议、风险提示。
- 在线预览和导出入口。
- 指标导入和账号洞察。
- 推荐链路模拟。

下一步重点不是增加更多说明文字，而是把“质量信号、风险原因、推荐瓶颈、导出状态”合并成运营能快速扫读的判断面板。

## 2. V0.3 产品目标

V0.3 的目标是建立“发布前质量门”：

```text
ASR 转写质量
  -> 候选队列质量
  -> 推荐链路模拟
  -> 人工审核/导出
  -> 手动发布
  -> CSV 数据回流
  -> SDK/API 回流 contract 预留
```

运营侧需要一眼看到：

- 当前节目是否适合批量导出。
- ASR 是否存在重复幻觉、广告口播或缺字幕。
- Top 候选是否以闭环结构为主，还是纯音频/弱上下文偏多。
- 哪些高分片段仍需要人工复核。
- 推荐模拟的瓶颈和实际导出动作怎么衔接。

## 3. 本轮已落地能力

新增只读质量洞察模块：

- `src/dso/quality/insights.py`
- `GET /videos/{video_id}/quality?top_k=30`

该接口输出：

- `health`：节目级质量分、等级和首要问题。
- `transcript`：ASR 来源、后端、片段数、后处理版本、重复幻觉数、广告口播数。
- `queue`：候选数、已评分数、导出数、Top 队列均分、平均时长、广告风险、弱上下文、闭环数量和类型分布。
- `issues`：可解释的问题列表。
- `actions`：下一步操作建议。
- `watchlist`：需要人工复核的候选片段。

真实样本 `video_f62d5de2297c4304` 当前结果：

- `source = whisper_cpp:base`
- `health.score = 62`
- `health.level = warn`
- 首要问题：ASR 幻觉/重复文本。
- Top10 中有品牌/导流口播和弱上下文候选。

## 4. 下一轮建议

### 4.1 Dashboard 接入质量洞察

在左侧视频/状态栏或推荐模拟页上方新增“质量哨兵”区域：

- 健康分和状态徽标。
- ASR 后端、重复幻觉、广告口播、候选闭环率。
- 3 条动作建议。
- Watchlist 快速跳转候选详情。

不要把它做成独立营销页；它应该是运营审核工作台里的发布前门禁。

### 4.2 ASR 质量护栏

不改变 ASR Agent 当前实验文件的前提下，产品侧优先消费以下信号：

- 重复文本比例。
- 静音/音乐段 hallucination。
- 广告口播词表。
- `postprocess_version`。
- whisper.cpp extra args 是否启用 VAD/过滤。

等 ASR Agent 确认稳定方案后，再把质量接口里的建议从“告警”升级为“阻断/允许导出”策略。

### 4.3 推荐模拟和质量联动

推荐模拟只回答“可能怎么分发”，质量洞察回答“素材是否可靠”。两者应该在 UI 上联动：

- 模拟高分但质量风险高：进入人工复核。
- 模拟高分且质量稳定：优先导出预览。
- 模拟低分但结构完整：进入标题/封面二次优化。
- 模拟低分且 ASR 风险高：等待 ASR 重跑后再评估。

### 4.4 抖音开放平台/SDK 接入路线对齐

现阶段如果完全依赖 CSV，系统只能做人工反馈闭环，无法稳定获得账号下已发布视频的长期表现、窗口指标和视频 ID 映射。下一轮需要把抖音开放平台/SDK 接入纳入数据回流主线，但定位应是“只读数据同步优先、自动发布延后”。

为避免和当前迭代冲突，SDK/OpenAPI 不插队替代 V0.3.1 的质量门、导入可信度和复盘面板，也不挤占 V0.4 的 ASR 二次转写、候选详情和人工复核工作流。接入路线按三段推进：

**V0.3.1：只预留 contract，不接真实平台**

- 在指标导入报告中明确 `sample_source`、`link_status`、`trust_status`，保证未来 CSV/API 两种来源能共用训练资格判断。
- 保持 CSV 是唯一生产可用回流路径。
- 不新增 OAuth、token、平台请求、后台同步任务。

**V0.4：运营工作台内补平台映射和同步状态壳**

- 账号授权状态占位：记录抖音账号、授权状态、scope、token 过期时间和最后同步时间字段，但不执行真实 OAuth 跳转、不保存真实 token。
- 平台视频映射：把 `candidate_segment_id`、`slice_variant_id`、`publishing_experiment_id` 和平台侧 `item_id/video_id` 绑定。
- 指标字段映射：将平台返回的播放、点赞、评论、分享、平均播放时长等字段映射到 `performance_metrics`；平台不提供的字段继续允许 CSV 补齐。
- 同步任务与审计：记录每次拉取的窗口、状态、错误、原始响应摘要和入库条数，避免重复写入和静默失败。
- UI 入口：在指标导入区域增加“绑定平台视频/同步表现数据”的状态位，但保留 CSV 导入作为兜底路径。
- 该阶段只用 fake client / mock response 验证字段映射，不发真实平台请求。

**V0.5：在可学习排序地基里落真实只读接入**

- OAuth 授权与 token refresh。
- 拉取授权账号视频列表，支持手动匹配和半自动匹配已导出的切片。
- 按 6h/24h/72h/7d/30d 窗口同步表现数据，写入现有 `metric_snapshots`、`training_samples` 和 `account_baselines`。
- 使用 fake client 和录制样例覆盖测试，不让核心反馈逻辑依赖真实平台环境。
- 权限不足、接口限流、账号解绑、视频删除等状态要显式暴露给运营，而不是吞掉后继续生成误导性洞察。

## 5. 版本边界

V0.3 不做：

- 自动发布。
- 无人值守账号操作。
- 刷量或模拟互动。
- 绕过平台规则。
- 重写 ASR/候选/评分核心算法。

V0.3 做：

- 把多 Agent 已验证的结论产品化。
- 让质量、推荐、导出和反馈数据在工作台里形成闭环。
- 为抖音开放平台/SDK 的只读数据回流预留数据 contract；真实接入后移到 V0.5。
- 保持所有新能力可测试、可解释、可回滚。
