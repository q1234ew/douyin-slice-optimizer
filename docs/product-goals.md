# 产品北极星目标与持续迭代原则

生效日期：2026-07-18

文档属性：规范性长期目标

适用范围：产品、算法、模型、数据、前端、后端、部署和评测

本文档定义 Douyin Slice Optimizer 的最高层目标和持续迭代约束。历史计划、实验报告或局部实现如果与本文档冲突，应先回到这里确认目标，再更新实现与文档。工程验收细则见 [development-requirements.md](development-requirements.md)，当前完成度见 [current-state.md](current-state.md)。

## 1. 产品使命

把自有或已授权的视频素材转化为可解释、可审核、可回测的高潜短视频候选，帮助运营提高在抖音等短视频平台获得优质流量、转发和关注的概率。

系统优化的是“成功概率与实验效率”，不承诺爆款，不模拟平台内部推荐系统，不刷量、不诱导互动、不绕过平台规则，也不自动替代人工发布终审。

## 2. 三项目标成果

### G1：已切短视频的筛选与推荐排名

系统必须支持单条或批量导入已经完成剪辑的短视频，并把原片段作为不可变候选进入统一排名流程，不要求再次切片。

标准流程：

```text
已切短视频
  -> 素材校验与去重
  -> ASR / OCR / 画面 / 音频 / 结构特征
  -> 内容分类与质量风险检查
  -> 账号、平台和历史样本匹配
  -> 多目标排序
  -> Top-K 推荐、解释和复核建议
  -> 人工发布实验
  -> 6h / 24h / 72h / 7d 指标回流
```

完成标准：

- 支持一条和一批短片导入，保留来源、原始边界、版本和内容 hash。
- 对同批候选输出稳定可复现的排名、分项得分、证据、风险、置信度和模型版本。
- 能回答“为什么推荐、适合哪个账号/场景、主要风险是什么、下一步应复核还是发布实验”。
- 新排序策略必须在冻结数据集上与当前基线比较，不能只展示主观案例。
- 发布数据回流后，能按曝光、观看、互动和关注指标复盘排名质量。

### G2：完整节目的智能切片、分类与排名

系统必须支持对完整节目自动理解、召回、切片、分类和排序，并将生成的短片候选复用 G1 的同一套筛选排名能力。

标准流程：

```text
完整节目
  -> ASR / OCR / 镜头 / 音频 / 节目结构理解
  -> 多策略候选召回
  -> 语义、静音、起音和切镜边界吸附
  -> 片段类型与叙事角色分类
  -> 进入 G1 统一排名流程
  -> 人工审核、预览、导出和指标回流
```

完成标准：

- 至少覆盖歌曲高光、开场钩子、人物故事、冲突/反转、点评、现场反应和情绪回报等可扩展类别。
- 候选边界有确定性时间轴证据；大模型建议不能无证据直接改写最终边界。
- 召回和排序分开评估：召回看 Gold `Recall@K`，排序看 `NDCG@K`、Top-K lift 和高/低互动样本区分能力。
- 模型不可用时仍能返回规则与历史先验基线，不能让整个切片流程失效。
- 已切短片和自动生成短片必须进入同一个 ranker contract，避免维护两套相互漂移的评分逻辑。

### G3：本地模型与公网大模型 API 的可控协同

系统默认本地优先，同时允许通过可插拔 Provider 访问公网模型 API，例如 Qwen、Kimi 或后续经过评估的多模态模型。具体厂商不是架构前提，调用能力、成本、隐私和降级 contract 才是长期接口。

每个 Provider 至少需要统一记录：

- 厂商、模型、接口版本、请求类型和提示词版本。
- 输入的视频秒数、帧数、图片数、音频秒数、文本/token 规模和请求字节数。
- 输出 token、延迟、重试、错误、限流、缓存命中和估算费用。
- 数据是否允许离开本地、授权依据、脱敏策略和保留期限。
- 本地基线、API 结果、融合结果以及最终采用原因。

默认路由顺序：

```text
确定性特征与缓存
  -> 本地轻量模型
  -> 仅对高价值或高不确定候选调用公网 API
  -> 结果校验与低权重融合
  -> API 失败时回退本地基线
```

公网模型不得因为“更大”就直接进入生产排序。必须先在冻结评估集运行 Shadow 对比，证明相对当前基线有稳定增益，并同时报告增量成本、端到端延迟和失败率。

## 3. 优化目标与指标口径

项目目标不是单独最大化播放量，而是优化一个受成本、风险和数据可信度约束的多目标结果：

```text
目标价值 = 优质流量 + 观看质量 + 转发价值 + 关注转化
         - 负反馈 - 合规风险 - 推理与运营成本
```

指标按优先级分为四层：

| 层级 | 核心指标 | 使用原则 |
| --- | --- | --- |
| 线上结果 | 曝光归一化播放、5 秒留存、平均观看比例、完播、复播、分享率、关注转化率、负反馈率 | 有 impressions 时优先使用率指标，避免把曝光规模误当内容质量 |
| 业务代理 | 收藏、评论质量、点赞、可见互动热度、账号历史匹配 | 播放或曝光缺失时只能标注为代理指标，不得宣称已证明高流量 |
| 离线算法 | Recall@K、NDCG@K、Top-K lift、严重错判率、覆盖率、校准误差 | 必须绑定冻结数据、时间切分/账号切分和基线版本 |
| 工程成本 | 单节目/单候选费用、GPU/CPU 时间、P50/P95 延迟、缓存命中率、失败率、人工复核时间 | 质量提升必须同时给出资源与运营开销 |

当播放量或曝光不可得时，系统应明确显示“高互动代理”或“研究结论”，不能把点赞、收藏或其他可见计数改名为流量。

## 4. 持续迭代原则

1. **目标对齐**：每项功能、模型或重构必须明确服务 G1、G2 或 G3，并说明影响的目标指标；无法建立映射的工作不进入主线优先级。
2. **双入口、一条排序链路**：已切短片和完整节目只在候选产生方式上不同，候选标准化之后共用分类、排序、解释、审核和回流 contract。
3. **证据优先**：模型升级先提出可证伪假设，再做冻结基线、消融、回测和 Shadow；单条好案例不能作为上线依据。
4. **防止数据泄漏**：排序评估优先使用时间切分、账号隔离或节目隔离；同源重复片段不能同时进入训练和验证。
5. **本地优先、API 按需**：可缓存、可确定性完成的步骤不调用公网 API；昂贵模型只处理高价值、高分歧或高不确定样本。
6. **成本可见**：任何新增模型必须报告质量、速度、显存/内存、API 费用和人工开销，不能只报告准确率。
7. **可降级**：外部模型、GPU 或网络不可用时保留本地基线结果，并清楚显示降级状态，不生成伪造的模型分。
8. **人工终审**：模型不自动写入人工 Gold、不自动批准高风险内容、不自动发布；生产权重变更需要可追溯审核。
9. **版本化与可回滚**：特征、提示词、模型、权重、数据集、报告和缓存必须带版本；新版本未过门禁时不覆盖当前生产基线。
10. **先闭环再复杂化**：优先补齐导入、理解、排名、审核、实验和回流闭环；在数据和基线不足时不急于引入复杂深度模型。
11. **研究账号不等于发布账号**：外部采集账号只提供跨账号研究先验；目标发布账号必须显式指定并用自身授权结果校准。目标结果缺失时保持冷启动，不把研究账号表现或模糊可见计数写成账号个性化证据。

## 5. 模型与算法准入门禁

新模型或新算法进入默认链路前，至少回答：

1. 它解决哪个已量化问题，当前基线是多少？
2. 使用哪份冻结数据，是否存在同源、时间或账号泄漏？
3. 对 Recall、NDCG、Top-K lift、严重错判或校准中的哪一项有提升？
4. 相对本地基线增加多少延迟、显存、费用和失败风险？
5. 哪些样本应该调用，哪些样本不值得调用？
6. 失败时如何降级，旧版本如何回滚？
7. 结果是 `research_only`、`shadow`、`candidate` 还是允许进入生产权重？

若样本量不足、质量提升不稳定或成本不可接受，应保留为研究候选，不因模型热度提前上线。

## 6. 公网 API 成本与安全门

- 默认关闭公网上传；只有用户配置 Provider、密钥、预算和数据许可后才启用。
- API key 只从环境变量或密钥服务读取，不写数据库、日志、缓存或前端 bundle。
- 每次任务支持单请求、单批次和单日预算；达到预算立即停止新增请求并回退本地结果。
- 对视频优先抽取代表帧、音频窗口和结构化摘要，不默认上传完整节目。
- 相同内容 hash、模型、提示词和参数应命中缓存，避免重复计费。
- 报告必须同时展示“质量增益、增量成本、单位有效候选成本和 P95 延迟”。
- 厂商返回内容视为不可信输入，需要 schema 校验、超时、重试上限和提示词回显/幻觉过滤。

## 7. 新算法与模型的主动汇报机制

开发过程中发现可能改善流量、转发、关注、切片召回、排序质量或成本效率的新算法/模型时，应执行以下动作：

1. 在 [model-and-algorithm-radar.md](model-and-algorithm-radar.md) 新增或更新条目。
2. 在当次工作结论中主动告知用户，不等待用户再次询问。
3. 用一页以内说明问题、候选方案、公开/本地证据、预期收益、成本、风险、集成位置和最小验证方案。
4. 明确建议为“立即验证、进入观察、暂不采用”之一。
5. 未完成冻结 benchmark 前，不把研究判断写成已证实收益。

## 8. 每轮迭代的最小交付说明

每轮迭代结束至少说明：

- 对齐的目标编号：G1 / G2 / G3。
- 用户可感知的结果和未完成部分。
- 使用的数据、基线、指标和验证结果。
- 新增的本地/API 计算成本与失败降级方式。
- 是否改变生产排序、人工 Gold、导出或发布行为。
- 发现的新算法/模型机会及其雷达状态。
